# Конституция проекта Solar Balls

> Это набор нерушимых принципов. Каждая спецификация, план и задача обязаны им соответствовать. Если принцип мешает — не обходите его, а правьте конституцию через PR с обоснованием.

**Версия:** 1.0.0
**Последнее обновление:** 2026-04-23

---

## Принцип 1. Спека — первична

Код без спецификации не пишется. Для каждой фичи в `specs/NNN-slug/` должны существовать:
- `spec.md` — **что** и **зачем** (без упоминания технологий);
- `plan.md` — **как** (технические решения, привязанные к стеку из конституции);
- `tasks.md` — пошаговый чек-лист реализации.

Перед началом кодинга все три файла прошли ревью и помечены статусом `Approved`.

## Принцип 2. pnpm-монорепо

Единственный пакетный менеджер — `pnpm`. Структура жёсткая:
- `apps/web` — Phaser-клиент.
- `apps/api` — Fastify-сервер.
- `packages/shared` — Zod-схемы и DTO, единственный источник типов.
- `content/` — игровой контент в виде JSON-файлов.

Никакого `npm`, `yarn` и локальных пакетов вне `packages/`.

## Принцип 3. Данные-ориентированный контент

Защитники, враги, уровни, товары магазина, достижения — только JSON-файлы в `content/`, валидируемые **Zod-схемой** из `packages/shared`. Добавление нового юнита = один JSON + один спрайт; правки кода быть не должно. Если фича требует нового поведения — расширяется `discriminatedUnion` в схеме, а не хардкодится в классе.

## Принцип 4. Shared-first типы

Любой тип, пересекающий границу клиент↔сервер или код↔контент, живёт в `packages/shared`. Дублирование типов в `apps/web` и `apps/api` запрещено. DTO входа/выхода API — тоже Zod-схемы, из которых выводятся TS-типы.

## Принцип 5. Универсальный ввод

Весь пользовательский ввод — через **Phaser Pointer Events** (`pointerdown`/`pointermove`/`pointerup`, `setDraggable`, `drag*`/`drop`). Отдельных веток «для мыши» и «для тача» быть не должно. Mobile-first hitbox ≥ 64 px.

## Принцип 6. Deploy-first

Проект разворачивается публично с **этапа 0**. Каждая ветка перед merge проходит CI и деплой в staging. Не существует «сначала допилим локально, потом задеплоим». `pnpm deploy:prod` должен работать всегда.

## Принцип 7. Играбельное демо на каждом этапе

Каждая спецификация обязана завершаться **демо, в которое можно поиграть**. «Невидимые» рефакторы без видимого демо требуют отдельного обоснования в `spec.md → Demo`.

## Принцип 8. Guest-first авторизация

Игра доступна без регистрации. На первом заходе пользователь автоматически получает **гостевой аккаунт** (cookie + серверный `GuestUser`). Реальная авторизация добавляется позже и умеет мержить прогресс гостя в аккаунт.

## Принцип 9. AI-ассеты в едином стиле

Если ассета нет — он генерируется в рамках того же PR, в который делается фича. Единый промпт-стиль: *«cartoonish neon space, vivid purples/cyans/magenta, rounded shapes, kawaii faces, Solar Balls style, transparent background 512×512 PNG»*. Файл кладётся в `content/images/`, оптимизируется до ≤ 200 КБ.

## Принцип 10. Тестируемые критерии приёмки

В каждой `spec.md` есть раздел *Acceptance Criteria* с проверяемыми пунктами («после размещения Солнца энергия увеличивается на 25 каждые 7 с»). Пункты не должны начинаться с «должно быть удобно/красиво/быстро» без числовой метрики.

## Принцип 11. Одна фича — одна ветка — один PR

Ветка `feature/NNN-slug` привязана к `specs/NNN-slug/`. PR в `main` допускается только с прохождением CI и обновлёнными `tasks.md` (все задачи закрыты) и `quickstart.md` (если фича меняет способ запуска).

## Принцип 12. Константы — в конституции, баланс — в контенте

Числа геймдизайна (урон, HP, кулдауны) живут в JSON контента. Технические константы (размер поля 5×9, виртуальное разрешение 1280×720, макс. размер PNG 200 КБ) живут здесь и меняются через PR в конституцию.

---

## Поправки

Конституция меняется только через PR с меткой `constitution`. В описании PR — причина и список затронутых спецификаций, которые нужно пересмотреть.
