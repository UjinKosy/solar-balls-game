# Solar Balls — Шаранутый Космос

PvZ-подобная игра в сеттинге «Шаранутого Космоса». Планеты-защитники отбивают атаку метеоритов, комет и чёрных дыр на поле 5×9 клеток.

- **Клиент:** Phaser 3 + TypeScript (Vite).
- **Сервер:** Fastify + TypeScript + Prisma.
- **БД:** PostgreSQL.
- **Менеджер пакетов:** `pnpm` (монорепо).
- **Методология:** Spec-Driven Development (SDD).

> Проект пока на стадии нулевого этапа: есть только план, конституция и инфраструктура SDD. Код появится при выполнении первой спецификации.

## Spec-Driven Development: короткое объяснение

**SDD** — методология, при которой каждая фича проходит через три артефакта, прежде чем появится код:

1. **`spec.md` — что и зачем.** Пользовательские истории, функциональные требования, критерии приёмки. Ни строчки кода, ни одного имени фреймворка.
2. **`plan.md` — как.** Технические решения: какие файлы, какие схемы, какие API. Проверка соответствия конституции.
3. **`tasks.md` — пошагово.** Атомарный чек-лист: от Setup до Deploy.

Только после этого — реализация.

У проекта есть **конституция** ([`.specify/memory/constitution.md`](.specify/memory/constitution.md)) — 12 нерушимых принципов. Любой план обязан им соответствовать.

## Как начать

### Шаг 1. Прочитайте конституцию

В Cursor откройте [`.specify/memory/constitution.md`](.specify/memory/constitution.md) или вызовите slash-команду:

```
/constitution
```

Эти 12 принципов — основа всех будущих решений. Если какой-то кажется лишним или наоборот не хватает — меняйте **сейчас**, пока спек ещё нет.

### Шаг 2. Изучите мастер-план

Откройте [`docs/plans/solar-balls-plan.md`](docs/plans/solar-balls-plan.md). Там 14 этапов (0–13), каждый даёт играбельное демо. Мастер-план — это «уровень моря», а спеки — конкретные волны на нём.

### Шаг 3. Создайте первую спеку

В Cursor вызовите:

```
/specify Этап 0 — монорепо с Phaser + Fastify + Postgres, деплой-каркас, пустой холст открывается по публичному URL
```

Агент:
1. Прочитает конституцию и мастер-план.
2. Задаст уточняющие вопросы (скоуп, out-of-scope).
3. Создаст папку `specs/001-monorepo-and-deploy/` и файл `spec.md` по шаблону.

Отредактируйте `spec.md` вручную, пока критерии приёмки и сценарии не станут точными. Проставьте статус `Approved`.

### Шаг 4. Сгенерируйте технический план

```
/plan
```

Агент создаст `specs/001-monorepo-and-deploy/plan.md`, прогонит **Constitution Check**, предложит архитектурные решения, список файлов, Data Model (если есть БД), API-контракты (если есть бэкенд), список ассетов с промптами AI-генерации и `quickstart.md`.

Если `Constitution Check` провален — агент остановится и попросит выбор: переделать спеку или выпустить поправку к конституции.

### Шаг 5. Разбейте на задачи

```
/tasks
```

Агент сгенерирует `tasks.md` — атомарный чек-лист в строгом порядке: Setup → Contract → Content → Core → UI → Integration → Deploy.

### Шаг 6. Реализуйте

Переключитесь в **agent mode** (это требуется для записи в non-markdown файлы) и вызовите:

```
/implement
```

Агент:
1. Создаст ветку `feature/001-monorepo-and-deploy`.
2. Будет выполнять задачи по порядку, ставя галочки в `tasks.md`.
3. Между группами задач — показывать прогресс.
4. В конце — прогонит все AC из `spec.md`, соберёт билд, откроет PR.

### Шаг 7. Следующая фича

После merge PR первой фичи — возвращайтесь к шагу 3 с `/specify` для этапа 1 (главное меню), и так до этапа 13.

## Этап 0: локальная проверка

```bash
pnpm install
POSTGRES_PORT=55432 docker compose up -d postgres
pnpm dev
```

- Клиент: откройте `http://localhost:5173`.
- API health-check: `curl -i http://localhost:3000/health`.
- Публичный URL демо: добавьте ссылку после деплоя в PR (staging/prod).

Остановить локальное окружение:

```bash
POSTGRES_PORT=55432 docker compose down
```

## Быстрый пример одной фичи полностью

Для понимания — как выглядит работа над одной фичей от начала до конца:

```
Пользователь: /specify Этап 2 — игровое поле 5×9 и drag-and-drop карт на мобилке и десктопе
AI: задаёт вопросы, создаёт specs/003-grid-and-drag-drop/spec.md
Пользователь: правит spec.md, AC и сценарии становятся чёткими, ставит Approved

Пользователь: /plan
AI: читает spec.md, проверяет конституцию, создаёт plan.md + quickstart.md

Пользователь: /tasks
AI: создаёт tasks.md с T-001 ... T-050

Пользователь: переключается в agent mode, /implement
AI: делает git checkout -b feature/003-grid-and-drag-drop, выполняет задачи,
    коммитит по одной, после Integration — запускает dev и просит проверить на мобилке,
    открывает PR
```

## Правила для людей и ИИ

1. Никогда не коммитьте код, если нет `spec.md` + `plan.md` + `tasks.md`.
2. Не обходите конституцию — меняйте её через PR с меткой `governance`.
3. Баланс (HP, урон, кулдауны) — только в JSON в `content/`.
4. Новый защитник или враг — без правок кода: один JSON + один спрайт.
5. Один PR = одна спека = одна играбельная демка.
